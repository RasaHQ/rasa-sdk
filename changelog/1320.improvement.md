**Barge-in / interruption support**

When the user speaks over the assistant (barge-in), the voice channel can notify the action server of exactly which chunks were heard before the interruption. The action server then records only those chunks to the tracker.

New `CollectingDispatcher` API:

- `cancel_stream()` — signals that the user interrupted. Subsequent `stream_chunk()` calls are silently dropped. The action continues running so it can still return Rasa events (slot sets, etc.).
- `acknowledge_heard_chunks(last_heard_chunk_index)` — records which delivered chunks the user actually heard. Called by the transport layer on receipt of `AckStreamChunks`.
- `is_streaming_cancelled` property — `True` after `cancel_stream()` has been called. Action authors can check this to optionally suppress side effects after a barge-in.
- `stream_end()` now always writes chunks to `dispatcher.messages` (and therefore to the tracker) on streaming transports. When cancelled it writes only the heard chunks (or, if no ack arrived, the delivered chunks); on the normal path it writes all accumulated chunks.

New gRPC RPC:

- `AckStreamChunks(StreamChunkAck) returns (google.protobuf.Empty)` — the voice channel calls this with the `response_id` of the active streaming response and `last_heard_chunk_index`. The action server looks up the dispatcher in its registry, records the heard slice, and sets the cancellation flag. `WebhookStream` detects the flag on the next iteration, drains the queue until the action finishes, and yields `final_result` (containing only the heard responses and all Rasa events) to the client.

The `response_id` field on `ChunkStart`, `Chunk`, and `ChunkEnd` events acts as the correlation key between the outbound stream and the inbound `AckStreamChunks` call.
